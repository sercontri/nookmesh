# GeoJSON Endpoints

NookMesh exposes location data through protected GeoJSON endpoints for consumption by compatible clients.

The primary client currently documented is:

```text
Guru Maps
```

although any GeoJSON-compatible consumer could be integrated.

---

## Purpose

GeoJSON endpoints allow NookMesh to:

- expose filtered location data
- apply per-user authentication
- enforce visibility rules
- deliver enriched location data
- decouple backend and client

---

## Format

Primary format:

```text
GeoJSON
```

Base structure:

```json
{
  "type": "FeatureCollection",
  "features": [...]
}
```

Each response contains only the data authorized for the authenticated user.

---

## Main endpoint

Current endpoint:

```text
/nookmesh.geojson
```

Example:

```text
https://geojson.yourdomain.com/nookmesh.geojson?token=TOKEN
```

The exact URL depends on your deployment.

---

## Authentication

Access requires an individual API token.

Example:

```text
?token=TOKEN
```

The API:

- validates authentication
- identifies the associated user
- loads runtime configuration
- applies visibility rules
- returns only authorized data

---

## What it actually returns

The API does not directly expose the raw GeoJSON generated by the worker.

It first applies:

- authentication
- visibility rules
- contextual filtering
- additional operational filters

Then it builds the final GeoJSON response for the client.

---

## Current response model

The current implementation generates two features for each visible entity:

### Icon feature

Primary visual representation.

Example:

```json
{
  "type": "Feature",
  "geometry": {
    "type": "Point",
    "coordinates": [-1.842799, 38.988115]
  },
  "properties": {
    "render": "icon",
    "icon_svg": "<svg>...</svg>",
    "name": "SANDRA",
    "desc": "Recent position..."
  }
}
```

---

### Label feature

Independent visual label.

Example:

```json
{
  "type": "Feature",
  "geometry": {
    "type": "Point",
    "coordinates": [-1.842799, 38.988115]
  },
  "properties": {
    "render": "label",
    "tid": "SA",
    "name": "SANDRA",
    "desc": "Recent position..."
  }
}
```

This enables:

- dynamic SVG iconography
- decoupled labels
- advanced visual control through MapCSS

---

## Current properties

The final response may include:

### In icon features

- `render`
- `icon_svg`
- `name`
- `desc`

---

### In label features

- `render`
- `tid`
- `name`
- `desc`

## Applied filtering

Before returning data, the API evaluates:

- valid token
- authenticated user
- shared groups
- `oculto_para` rules
- maximum age
- viewer self-exclusion (if configured)
- proximity filtering
- multi-device rules

---

## Visibility filtering

The core logic works as:

```text
shared groups - hidden exclusions
```

Only users with valid shared groups are shown.

---

## Cases that return an empty result

The API may return an empty GeoJSON if:

### User has no groups

The authenticated user has no groups defined in:

```text
visibility.json
```

---

### Viewer has no recent position

If:

```env
REQUIRE_RECENT_VIEWER_POSITION_FOR_PROXIMITY=true
```

and the viewer has no recent position:

```text
<= MAX_EDAD_MIN
```

the response will be empty.

---

### No visible users

This may happen due to:

- visibility rules
- hidden users
- stale data
- proximity filtering

---

## Metadata sanitization for non-staff users

Users without the role:

```text
staff
```

currently receive a reduced version of certain metadata.

The API removes lines such as:

- device information
- connection type
- visible groups

This reduces exposure of internal details.

---

## Dynamic SVG generation

Icons do not depend entirely on the client.

The API dynamically generates SVG based on:

- role
- freshness
- heading

This enables consistent rendering even on clients with limited styling support.

## Client caching

Some clients, especially map applications, may cache responses.

This may affect:

- refresh frequency
- real-time perception
- visual updates

This is not directly controlled by the endpoint itself.

---

## Troubleshooting

### Endpoint does not respond

Check:

- API availability
- API container
- reverse proxy
- DNS
- HTTPS
- connectivity

---

### Access denied

Check:

- correct token
- valid token
- regenerated configuration

If you modified users:

```bash
./auth/generate.sh
```

---

### Empty GeoJSON

Check:

- recent data availability
- viewer groups
- `visibility.json`
- operational filters
- viewer recent position
- proximity rules

---

### Guru Maps shows nothing

Check:

- exact URL
- correct token
- GeoJSON format
- overlay
- MapCSS style
- client cache
