# Authentication

🇪🇸 [Versión en español](authentication.es.md)

NookMesh protects access to location data through per-user authentication and backend validation.

The goal is to prevent accidental exposure or unauthorized access to sensitive location information.

Unlike models where the client decides what to display, NookMesh ensures that the API fully controls which data each consumer can receive.

---

## Purpose

Authentication protects:

- locations
- associated metadata
- API endpoint access
- user isolation
- secure enforcement of visibility rules

It provides:

- granular access control
- selective credential revocation
- real privacy
- separation between consumers

---

## Current model

The API currently uses individual token-based authentication.

Example:

```text
https://geojson.yourdomain.com/nookmesh.geojson?token=TOKEN
```

Each user not marked as:

```json
"system_user": true
```

receives their own API access token.

---

## API tokens

Tokens are automatically generated by:

```bash
./auth/generate.sh
```

and stored in:

```text
config/generated/api-tokens.txt
```

Format:

```text
user:token
```

Example:

```text
sergio:0123456789abcdef...
```

---

## Generation and persistence

Current behavior:

- if a new user appears in `config/users.json`, a token is automatically created
- if the user already exists, their token is preserved
- if `regen_token: true` is set, a new token is generated
- if the user is removed from `users.json`, their token disappears

This avoids unnecessarily breaking already configured clients.

---

## Token expiration

Currently:

```text
tokens do not expire automatically
```

They remain valid until:

- manual regeneration
- user removal
- explicit recreation

---

## Authentication flow

Simplified real flow:

### 1

The client requests a protected endpoint.

Example:

```text
/nookmesh.geojson?token=TOKEN
```

---

### 2

The API validates the provided token.

---

### 3

If the token is valid:

- identifies the associated user
- loads their visibility configuration
- applies access rules
- filters the requested GeoJSON

---

### 4

If validation fails:

access is denied.

---

## Authentication does not imply full access

Successfully authenticating does not mean seeing all data.

After identity validation, NookMesh still applies:

- shared groups
- `oculto_para` exclusions
- operational filters
- contextual restrictions

Example:

an authenticated user may still be unable to see certain locations.

---

## Security

Authentication is enforced entirely in the backend.

It is never delegated to the client.

This prevents:

- client-side tampering
- accidental exposure
- visibility rule bypass

The client only receives the final authorized result.

---

## Token revocation

If a credential is compromised, set:

```json
"regen_token": true
```

for that user and run:

```bash
./auth/generate.sh
```

This:

- generates a new token
- immediately invalidates the previous one
- updates operational credentials
- restarts compatible services if running

Afterward, the system automatically resets:

```json
"regen_token": false
```

---

## HTTPS

Very important.

Because the token currently travels in the URL:

```text
?token=TOKEN
```

you should always use:

```text
HTTPS
```

Without encryption, credentials may be exposed.

---

## Query string considerations

Using tokens in query parameters is currently maintained for compatibility with clients such as:

```text
Guru Maps
```

but this has security implications.

Tokens may appear in:

- HTTP logs
- reverse proxies
- browser history
- monitoring tools
- debugging traces

Therefore:

- always use HTTPS
- limit log exposure
- secure intermediary infrastructure

---

## Best practices

### Do not share tokens

Each token identifies a specific user.

---

### Rotate tokens if compromised

If a token may have leaked:

regenerate it immediately.

---

### Apply least privilege

Do not grant unnecessary access.

---

### Protect intermediary infrastructure

Especially:

- reverse proxies
- logs
- observability systems
- debugging environments

---

## Users without tokens

Users marked as:

```json
"system_user": true
```

do not receive API tokens.

Typical example:

```text
recorder
```

These users exist for internal infrastructure purposes.

Not for visual consumption.

---

## Protected resources

The same model can be used to protect other API-served resources.

Examples:

- GeoJSON
- visual assets
- MapCSS styles
- auxiliary endpoints

---

## Current limitations

There is currently no support for:

- HTTP header authentication
- bearer tokens
- automatic expiration
- refresh tokens
- SSO integration

The current model prioritizes simplicity and compatibility with existing clients.

---

## Possible future evolution

Potential future models:

- Authorization headers
- bearer tokens
- signed requests
- auth proxy
- SSO
- short-lived credentials

---

## Relationship with visibility

Authentication answers:

```text
Who are you?
```

Visibility answers:

```text
What are you allowed to see?
```

They are separate but complementary layers.

---

## Relationship with Guru Maps

Guru Maps only consumes authenticated endpoints.

It does not participate in:

- authentication
- authorization
- internal visibility decisions
