# Data Flow

🇪🇸 [Versión en español](data-flow.es.md)

This document describes the complete journey of a location update from the user's device to its final visualization in the client.

---

## Visual summary

```text
OwnTracks
   ↓
MQTT publish
   ↓
MQTT Broker (Mosquitto)
   ↓
OwnTracks Recorder
   ↓
Persistence
   ↓
GeoJSON Worker
   ↓
Protected API
   ↓
Guru Maps / compatible client
```

---

## Complete flow

## 1. Location capture

The device obtains a position using GPS and operating system sensors.

Typical information:

- latitude
- longitude
- accuracy
- altitude
- speed
- heading
- battery
- connectivity state
- timestamp
- tracker ID (`tid`)

Example:

```json
{
  "_type": "location",
  "lat": 38.561445,
  "lon": -0.212222,
  "acc": 13,
  "alt": 683,
  "vel": 10,
  "cog": 221,
  "batt": 80,
  "conn": "m",
  "tst": 1779701034,
  "tid": "SA"
}
```

---

## 2. MQTT publication

OwnTracks publishes the event to the MQTT broker.

Example:

```text
owntracks/sandra/iphone
```

At this point:

- the client must authenticate successfully
- the ACL must allow publishing
- connectivity must be operational

---

## 3. Reception at the MQTT broker

Mosquitto receives the message and validates:

- MQTT username
- password
- ACL
- allowed topic

If validation succeeds:

the message is distributed to authorized subscribers.

---

## 4. Consumption by Recorder

OwnTracks Recorder consumes the MQTT message.

Responsibilities:

- parse the OwnTracks payload
- persist the event
- maintain latest positions per device

At this stage there is no:

- visibility filtering
- API authentication
- visual logic

---

## 5. Persistence

Data is stored in:

```text
data/owntracks/store/
```

Including:

```text
data/owntracks/store/last/
```

Real example:

```text
data/owntracks/store/last/sergio/iphone/sergio-iphone.json
```

This persistence acts as the operational source for the downstream pipeline.

---

## 6. Runtime configuration

NookMesh maintains automatically generated runtime artifacts.

Relevant files:

```text
config/generated/api-tokens.txt
data/runtime/visibility.json
```

Generated by:

```bash
./auth/generate.sh
```

These determine:

- API authentication
- valid users
- groups
- hidden relationships
- roles

---

## 7. Worker processing

The worker reads persisted latest positions and builds the base GeoJSON.

Inputs:

```text
data/owntracks/store/last/
config/filtros.env
data/runtime/visibility.json
```

Typical processing:

- latest position lookup
- age calculation
- connectivity interpretation
- movement interpretation
- property enrichment
- contextual description generation
- stale location removal

---

## 8. Base GeoJSON generation

The worker generates:

```text
data/public/nookmesh.geojson
```

Format:

```json
{
  "type": "FeatureCollection",
  "features": [...]
}
```

This GeoJSON contains enriched information but is not yet filtered for the requesting user.

It may include:

- name
- tid
- timestamps
- age
- heading
- groups
- role
- accuracy
- visual stroke
- enriched description

---

## 9. Client request

A client performs an authenticated request.

Example:

```text
https://geojson.yourdomain.com/nookmesh.geojson?token=TOKEN
```

---

## 10. API authentication

The API validates:

- API token
- user associated with the token
- existence in runtime configuration

If authentication fails:

access is denied.

---

## 11. Visibility resolution

The API determines:

- viewer groups
- evaluated user groups
- `oculto_para` exclusions

Conceptual model:

```text
shared_groups - hidden_for
```

If no visible groups exist:

the location is excluded.

---

## 12. Contextual operational filters

The API may additionally apply:

- self-viewer exclusion (`EXCLUDE_VIEWER_IN_OUTPUT`)
- proximity filtering (`EXCLUDE_NEARBY_METROS`)
- viewer recent-position validation
- multi-device merging (`MERGE_CLOSEST_DEVICES`)

Important case:

if:

```env
REQUIRE_RECENT_VIEWER_POSITION_FOR_PROXIMITY=true
```

and the viewer does not have a recent position:

the API returns an empty result.

---

## 13. Final GeoJSON construction

The API generates the final client-consumable GeoJSON.

It transforms each visible entity into multiple visual features.

Conceptual example:

### Icon

```json
"render": "icon"
```

Includes:

- dynamic SVG
- descriptive tooltip

---

### Label

```json
"render": "label"
```

Includes:

- `tid`
- visual text

---

Result:

GeoJSON specifically built for that authenticated viewer.

---

## 14. Client rendering

Guru Maps consumes the result and applies visual rendering.

Depending on the style:

- icons
- labels
- SVG
- tooltips
- conditional rendering

The client never decides access permissions.

It only renders the authorized result.

## Potential failure points

### No data arrives

Check:

- OwnTracks
- MQTT credentials
- connectivity
- broker ACLs

---

### Recorder does not persist

Check:

- MQTT subscription
- recorder logs
- recorder configuration
- mounts

---

### Base GeoJSON is not generated

Check:

- worker
- mounted paths
- permissions
- timestamps
- `MAX_EDAD_MIN`

---

### API returns empty data

Check:

- token
- visibility rules
- operational filters
- proximity filtering
- viewer without recent position

Regenerate if needed:

```bash
./auth/generate.sh
```

---

### Guru Maps shows no locations

Check:

- endpoint
- valid token
- connectivity
- GeoJSON format
- MapCSS style
- client cache
