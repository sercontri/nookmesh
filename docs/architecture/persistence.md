# Persistence and Storage

NookMesh separates code, configuration, and persistent operational data to maintain a modular, maintainable architecture suitable for real-world deployments.

---

## Purpose

Separating persistent storage allows:

- preserving state across restarts
- decoupling data from container lifecycle
- simplifying backups
- easing troubleshooting
- maintaining a modular architecture

---

## General structure

```text
nookmesh/
├── api/
├── auth/
├── config/
│   └── generated/
├── data/
│   ├── owntracks/
│   ├── public/
│   └── runtime/
├── docs/
├── mqtt/
├── recorder/
└── worker/
```

---

## Conceptual separation

### Code

System implementation:

```text
api/
auth/
mqtt/
recorder/
worker/
```

---

### Source configuration

Manually managed editable configuration:

```text
config/
```

Examples:

- users
- operational filters
- recorder configuration
- TLS certificates

Typical files:

```text
config/users.json
config/filtros.env
config/recorder.env
```

---

### Generated configuration

Automatically derived operational artifacts:

```text
config/generated/
```

Examples:

```text
mqtt-passwords.txt
mqtt-acl.txt
api-tokens.txt
```

These files should not be edited manually.

---

### Operational data

Persistence and runtime artifacts:

```text
data/
```

---

## data/owntracks

OwnTracks Recorder persistence.

This corresponds to the recorder’s operational storage.

Relevant path:

```text
data/owntracks/store/
```

Including:

```text
data/owntracks/store/last/
```

Real example:

```text
data/owntracks/store/last/sergio/iphone/sergio-iphone.json
```

May contain:

- latest locations per device
- internal recorder structures
- history maintained by OwnTracks Recorder

This is the primary source for the transformation pipeline.

---

## data/public

Client-consumable artifacts.

Current example:

```text
data/public/
├── nookmesh.geojson
└── nookmesh_v1.mapcss
```

---

### nookmesh.geojson

Automatically generated by the worker:

```text
data/public/nookmesh.geojson
```

Contains the enriched base GeoJSON used by the API.

It does not necessarily represent the final result delivered to the authenticated client.

---

### nookmesh_v1.mapcss

Editable visual asset served to compatible clients.

Example:

```text
data/public/nookmesh_v1.mapcss
```

Currently used by Guru Maps.

It is not generated automatically.

---

## data/runtime

Automatically derived runtime state.

Example:

```text
data/runtime/
└── visibility.json
```

Generated from:

```text
config/users.json
```

via:

```bash
./auth/generate.sh
```

Used for:

- groups
- exclusions
- roles
- optimized API filtering structure

This is not manual configuration.

---

## config/generated

Operational files automatically generated by:

```bash
./auth/generate.sh
```

Example:

```text
config/generated/
├── mqtt-passwords.txt
├── mqtt-acl.txt
└── api-tokens.txt
```

---

### mqtt-passwords.txt

Mosquitto-compatible password database.

Used for MQTT authentication.

---

### mqtt-acl.txt

Automatically generated MQTT ACLs.

Control MQTT read/write permissions per user.

---

### api-tokens.txt

Persistent API authentication tokens.

Format:

```text
user:token
```

Used for HTTP authentication.

---

## Persistence flow

Main pipeline:

```text
OwnTracks
   ↓
MQTT
   ↓
Recorder
   ↓
data/owntracks/store
   ↓
Worker
   ↓
data/public/nookmesh.geojson
   ↓
API
   ↓
Client
```

Configuration pipeline:

```text
config/users.json
   ↓
auth/generate.sh
   ↓
config/generated
   ↓
data/runtime
```

---

## Docker persistence

These paths should be persisted using bind mounts or volumes.

Purpose:

- prevent data loss
- allow safe upgrades
- decouple containers from data

Conceptually:

```text
host filesystem ↔ containers
```

---

## Backups

Recommended to include:

### Critical

Source configuration:

```text
config/
```

Location persistence:

```text
data/owntracks/store/
```

Especially:

```text
config/users.json
config/filtros.env
config/recorder.env
```

---

### Important depending on deployment

API tokens:

```text
config/generated/api-tokens.txt
```

If lost:

clients will need to be reconfigured.

---

### Regenerable

Can be regenerated:

```text
config/generated/mqtt-passwords.txt
config/generated/mqtt-acl.txt
data/runtime/visibility.json
```

---

### Versioned code

Recoverable from Git:

```text
api/
auth/
mqtt/
recorder/
worker/
docs/
```

---

## Git

Real data and secrets should not be versioned.

Examples:

```text
config/users.json
config/generated/*
data/owntracks/*
data/runtime/*
```

The repository may include safe placeholders or examples.

Examples:

```text
.gitkeep
example configs
public demo assets
```

## Troubleshooting

### Locations do not appear

Check:

```text
data/owntracks/store/last/
```

---

### Base GeoJSON is empty

Check:

```text
data/public/nookmesh.geojson
```

---

### Visual issues

Check:

```text
data/public/nookmesh_v1.mapcss
```

---

### Visibility issues

Check:

```text
data/runtime/visibility.json
```

---

### API authentication issues

Check:

```text
config/generated/api-tokens.txt
```

---

## Best practices

### Do not commit real data

Never upload:

```text
config/users.json
config/generated/
data/owntracks/
data/runtime/
```

---

### Do not edit runtime manually

Avoid modifying:

```text
config/generated/
data/runtime/
```

---

### Customize visuals intentionally

You may modify:

```text
data/public/nookmesh_v1.mapcss
```

if you want to alter visual rendering.

---

## Summary

NookMesh clearly separates:

- code
- source configuration
- generated configuration
- operational persistence
- derived runtime state
- client-consumable artifacts

This is part of the project's modular design.
