# Visual Customization with MapCSS

🇪🇸 [Versión en español](mapcss.es.md)

NookMesh allows visual customization of location rendering in compatible clients through **MapCSS**.

Currently, the primary client leveraging this capability is:

```text
Guru Maps
```

This makes it possible to transform functional GeoJSON into a rich, clear visualization adapted to different scenarios.

---

## Goal

MapCSS allows customization of:

- icons
- labels
- sizes
- offsets
- visual priorities
- conditional behavior
- overlap handling
- contextual presentation

This clearly separates:

```text
data
```

from

```text
visual presentation
```

## Current NookMesh visual model

The current implementation generates **two GeoJSON features for each visible entity**.

This allows iconography and text to be decoupled.

---

### Icon feature

Represents the main visual icon.

Example:

```json
"render": "icon"
```

Responsible for:

- primary iconography
- dynamic SVG
- contextual description

---

### Label feature

Represents the independent text label.

Example:

```json
"render": "label"
```

Responsible for:

- visible text
- short identifier (`tid`)
- contextual description

---

This allows separation of:

- icon
- text
- visual behavior

and provides greater rendering flexibility.

---

## Simplified real example

Icon feature:

```json
{
  "type": "Feature",
  "properties": {
    "render": "icon",
    "icon_svg": "<svg ...>",
    "name": "SANDRA",
    "desc": "Recent position..."
  }
}
```

Label feature:

```json
{
  "type": "Feature",
  "properties": {
    "render": "label",
    "tid": "SA",
    "name": "SANDRA",
    "desc": "Recent position..."
  }
}
```

## Relevant properties

### `render`

Defines the representation type.

Current values:

```text
icon
label
```

---

### `icon_svg`

Dynamic SVG generated by the backend.

Example:

```json
"icon_svg": "<svg ...>"
```

MapCSS does not build the icon.

It only decides how to render it.

---

### `tid`

Short identifier.

Example:

```json
"tid": "SA"
```

Typically used in labels.

---

### `name`

Logical user name.

Example:

```json
"name": "SANDRA"
```

---

### `desc`

Enriched contextual description.

May include:

- age
- accuracy
- speed
- battery
- altitude
- heading

Depending on the authenticated user's role, certain sensitive fields may be automatically hidden.

Non-`staff` users do not receive:

- device
- connection
- internal groups

---

## What the backend actually controls

A significant portion of the current visual logic happens in the backend.

Examples:

- SVG generation
- freshness-based colors
- heading marker
- contextual enrichment
- permission-based sanitization

This means not all visual logic depends on MapCSS.

---

## Current visual states

Currently:

```text
< 5 min      → green
5–29 min     → orange
30–60 min    → red
> MAX_AGE    → hidden
```

These colors are generated inside the dynamic SVG.

---

## Heading / direction

If valid heading data exists:

the backend generates an oriented black marker inside the SVG.

This allows movement direction to be displayed even with very simple MapCSS styling.

## What MapCSS actually controls

The style can control:

- showing or hiding labels
- icon sizes
- offsets
- displayed text
- visual priorities
- collisions
- conditional behavior
- selector-based rendering

While the backend provides:

- data
- SVG
- enriched properties
- contextual logic

---

## Conceptual examples

Show labels:

```css
node[render="label"] {
    text: eval(tag("tid"));
}
```

Show SVG icons:

```css
node[render="icon"] {
    icon-image: eval(tag("icon_svg"));
}
```

These examples are conceptual.

Guru Maps implements MapCSS support with its own specific behavior.

---

## Practical limitations

### Overlap

Guru Maps may hide elements when they overlap.

This can affect:

- labels
- icons
- complex combinations

---

### Zoom-dependent behavior

Some elements may behave differently depending on zoom level.

---

### Cache

Style changes may take time to appear because of client caching.

---

### Partial compatibility

Guru Maps does not implement the full standard MapCSS ecosystem exactly.

Some rules may behave differently.

---

## Troubleshooting

### I see icons but no labels

Check:

- `render="label"` selector
- `text` property
- visual collisions
- current zoom level

---

### Labels disappear

Possible causes:

- overlap
- visual priority
- conditional rules
- client behavior

---

### Icons do not appear

Check:

- `icon_svg` property
- `render="icon"` selector
- style accessibility

---

### Changes do not appear

Possible causes:

- Guru Maps cache
- incomplete reload
- outdated style URL

---

### Unexpected colors

Remember:

current colors come from backend SVG generation, not the style itself.
